# SOP Combine & Export — Design Spec
**Date:** 2026-04-27  
**Feature:** Combine 2+ SOP documents into one DOCX/PDF export (Parts-based, continuous step numbering)

---

## Problem

A single KT process is sometimes recorded in 2–3 separate sessions. Each session generates its own SOP in the platform. Infomate needs to export a **single professional document** that covers all parts of the process, instead of sending multiple separate files.

---

## Design Decisions

| Decision | Choice | Reason |
|---|---|---|
| Recordings stay separate? | Yes | Individual SOPs remain untouched in the system |
| New SOP entity created? | No | Combined doc is export-only — no DB record for the merged SOP |
| Structure of combined doc | Parts/Sections | Each recording becomes a labeled Part (Part 1, Part 2…) |
| Step numbering | Continuous across parts | Reads as one unified process (Step 1…7 → 8…14 → 15…20) |
| Who can combine? | Editor + Admin only | Use `require_editor` dependency — same as other write endpoints |
| Process map in combined doc | Omitted | Each SOP has its own process map; combining them is out of scope |
| `export_history` for combined | Skipped | No single `sop_id` to attach to; combined exports are ephemeral |
| Template for combined doc | New `sop_combined_template.docx` | Existing template is hardwired for single-SOP Jinja2 context; combined needs `parts[]` structure |
| Step offset logic | Overwrite `sequence` field directly | `_build_context` reads `step["sequence"]` — must overwrite, not add `display_sequence` |
| `_build_context` reuse | Not reused | `render_combined_sop` builds its own context from scratch — avoids TOC/process-map single-SOP assumptions |

## User Flow

```
Dashboard
  └─ "Combine SOPs" button (editor/admin only)
        │
        ▼
/combine  — 3-step wizard
  Step 1: Select SOPs
    - Checkbox-select from SOP list (min 2, max 10)
    - Shows: title, recording date, step count

  Step 2: Order & Label Parts
    - Drag to reorder selected SOPs
    - Rename each part label (default: "Part 1", "Part 2"…)
    - "Part label" appears as a bold section header in the combined doc

  Step 3: Preview & Export
    - Lists all steps grouped by part with continuous numbering
    - "Export DOCX" / "Export PDF" buttons
    - Triggers download of combined document
```

## API Contract

### `POST /api/sops/combine/export?format=docx`
**Auth:** `require_editor`

**Request body:**
```json
{
  "parts": [
    { "sop_id": "uuid-1", "label": "Part 1: Login & Setup" },
    { "sop_id": "uuid-2", "label": "Part 2: Processing & Approval" },
    { "sop_id": "uuid-3", "label": "Part 3: Export & Filing" }
  ],
  "title": "Aged Debtor Process — Combined"
}
```

**Logic:**
1. Load full SOPDetail for each `sop_id` (eager-load steps + sections)
2. Assign continuous step sequence numbers (overwrite `sequence` field):
   ```python
   offset = 0
   for part in parts:
       for step in part_sop.steps:
           step.sequence = step.sequence + offset  # overwrite, not new field
       offset += len(part_sop.steps)
   ```
3. Post to extractor `/api/render-combined`
4. Return `ExportResponse` (download_url, filename, format)
5. **No** `export_history` DB insert (no single sop_id)

**Response:** same `ExportResponse` shape as single-SOP export
```json
{
  "download_url": "https://...",
  "filename": "combined_<timestamp>.docx",
  "format": "docx"
}
```

## `render_combined_sop()` Logic

Builds its own Jinja2 context — does NOT call `_build_context`:

```python
def render_combined_sop(combined_id, fmt, title, parts, azure_blob_base_url, azure_sas_token):
    # Load sop_combined_template.docx
    tpl = DocxTemplate(COMBINED_TEMPLATE_PATH)

    # Build parts context — steps already have continuous sequence numbers
    parts_ctx = []
    for part in parts:
        steps_ctx = [_build_step_ctx(tpl, s, tmp_dir) for s in part["sop_data"]["steps"]]
        parts_ctx.append({"label": part["label"], "steps": steps_ctx})

    context = {
        "title": title,
        "today": date.today().strftime("%d %b %Y"),
        "parts": parts_ctx,
        # TOC: list of {label, step_from, step_to}
        "toc_entries": _build_combined_toc(parts),
    }

    tpl.render(context)
    # Save, upload to exports/combined/<combined_id>.docx, optionally convert to PDF
```

## `sop_combined_template.docx` Jinja2 Layout

Key template variables used:
- `{{ title }}` — combined SOP title
- `{{ today }}` — export date
- `{% for entry in toc_entries %}` — TOC rows
- `{% for part in parts %}` — part section headers + steps loop
- `{{ part.label }}` — e.g. "Part 1: Login & Setup"
- `{% for step in part.steps %}` — step rows (same fields as single-SOP template)

Build this template using `build_template.py` or by manually creating a `.docx` with docxtpl markers, following the same pattern as `sop_template.docx`.

## Frontend Route: `/combine`

**State:** `selectedIds: string[]`, `parts: CombinePartInput[]`, `wizardStep: 0|1|2`

**Step 0 — Select SOPs:**
- Fetches all SOPs via existing `fetchSOPs()`
- Checkbox on each SOP row — shows title, date, step_count badge
- "Next →" enabled when ≥ 2 selected

**Step 1 — Order & Label:**
- Drag-and-drop list (HTML5, same pattern as process map drag)
- Editable label input per part (default: "Part 1", "Part 2"…)
- "Next: Preview →"

**Step 2 — Preview & Export:**
- Fetches full SOPDetail for each selected SOP (`fetchSOP(id)`)
- Renders flat step list under bold part headers with continuous numbering
- "Export DOCX" / "Export PDF" → calls `exportCombinedSOPs(parts, title, format)`
- Shows spinner → then download link

---

## What's NOT in scope
- Merging or displaying process maps in the combined doc
- Saving the combined result as a new SOP entity in the DB
- Deduplication of steps across recordings
- Diff/compare view between recordings
- `export_history` record for combined exports
