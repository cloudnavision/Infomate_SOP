# Phase 1.5c: Backend JWT Validation + Role Guards

### Objective
Add JWT validation middleware to the FastAPI backend so all API endpoints require a valid Supabase Auth token. Create role-based dependency functions that can be applied per-route to enforce viewer/editor/admin permissions.

### Prerequisites
- Phase 1.5a complete — GET /api/auth/me endpoint exists
- Phase 1.5b complete — Frontend sends JWT in Authorization header

### What to Build

**Install Dependencies (in api/):**
Add to `requirements.txt`:
- `python-jose[cryptography]` — JWT decoding and validation
- `httpx` — async HTTP client to fetch Supabase JWKS (JSON Web Key Set)

**Environment Variables:**
Add to `.env` and `.env.example`:
```
SUPABASE_URL=https://<your-project-ref>.supabase.co
SUPABASE_JWT_SECRET=<your-supabase-jwt-secret>
```
The JWT secret is found in Supabase Dashboard → Settings → API → JWT Secret. This is used to verify token signatures.

**JWT Validation (api/app/middleware/auth.py):**

A module that validates Supabase Auth JWTs:

- Extract token from `Authorization: Bearer <token>` header
- Decode and validate using the Supabase JWT secret
- Verify: expiration, issuer (must be Supabase URL), audience
- Return the decoded payload containing `sub` (Supabase user ID), `email`, `exp`
- Raise `HTTPException(401)` if token is missing, expired, or invalid

**Auth Dependencies (api/app/dependencies/auth.py):**

FastAPI dependency functions that can be injected into route handlers:

`get_current_user(request, db)`:
- Calls JWT validation to get the decoded token
- Extracts `email` from the token
- Looks up the email in the `users` table
- If found → returns the `User` SQLAlchemy model instance
- If NOT found → raises `HTTPException(403, "Access denied — user not registered")`
- This is the core dependency — every protected route uses this

`require_role(minimum_role)`:
- Returns a dependency function that wraps `get_current_user`
- After getting the user, checks if their role meets the minimum requirement
- Role hierarchy: `admin` (3) > `editor` (2) > `viewer` (1)
- If user's role level >= required level → returns user
- If not → raises `HTTPException(403, "Insufficient permissions")`
- Usage: `current_user: User = Depends(require_role("editor"))`

**Convenience dependencies:**
- `require_viewer = require_role("viewer")` — any authenticated registered user
- `require_editor = require_role("editor")` — editor or admin
- `require_admin = require_role("admin")` — admin only

**Protect Existing Routes:**

Apply auth dependencies to all existing API endpoints:

| Endpoint | Guard |
|----------|-------|
| `GET /api/sops` | `require_viewer` |
| `GET /api/sops/{id}` | `require_viewer` |
| `GET /api/sops/{id}/steps` | `require_viewer` |
| `GET /api/sops/{id}/steps/{step_id}` | `require_viewer` |
| `GET /api/sops/{id}/sections` | `require_viewer` |
| `GET /api/sops/{id}/transcript` | `require_viewer` |
| `GET /api/sops/{id}/watchlist` | `require_viewer` |
| `GET /api/auth/me` | Special — validates JWT but checks users table itself |
| `GET /api/media/video/{sop_id}` | `require_viewer` |
| Future write endpoints | `require_editor` or `require_admin` as appropriate |

Implementation: add `current_user: User = Depends(require_viewer)` as a parameter to each route handler function. The dependency injection runs automatically before the handler.

**SOP Visibility Rule:**

Apply status-based filtering on the `GET /api/sops` endpoint based on role:
- **Viewer** → only sees SOPs with `status = 'published'`
- **Editor** → sees `published`, `draft`, `in_review`
- **Admin** → sees all statuses including `processing` and `archived`

This is implemented as a query filter in the route handler based on `current_user.role`.

**Update Config (api/app/config.py):**
Add `SUPABASE_URL` and `SUPABASE_JWT_SECRET` to the Settings model.

**Health Endpoint Exception:**
The existing health check endpoints (`/health`, `/api/health`) should remain public (no auth required) — they're used by Docker health checks and monitoring.
