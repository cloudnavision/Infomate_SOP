# Phase 1.5d: Admin User Management

### Objective
Build API endpoints and a frontend page for admins to manage platform users — add new users with email and role, change roles, and remove access. This is the mechanism that controls who can sign in (Option 3: pre-registered access).

### Prerequisites
- Phase 1.5c complete — Backend JWT validation and role guards working
- Admin user exists in users table (your own email with `admin` role)

### What to Build

**API Endpoints (api/app/routes/users.py):**

All endpoints require `require_admin` — only admins can manage users.

`GET /api/users` — List all users:
- Returns all users from the `users` table
- Response: array of `{ id, email, name, role, created_at, updated_at }`
- Optional query param `?role=viewer` to filter by role
- Ordered by `created_at DESC`

`POST /api/users` — Add a new user:
- Request body: `{ email: string, name: string, role: "viewer" | "editor" | "admin" }`
- Validate email format
- Check for duplicate email — return 409 Conflict if email already exists
- Insert into `users` table
- Return the created user record with 201 status

`PATCH /api/users/{user_id}` — Update user role:
- Request body: `{ role: "viewer" | "editor" | "admin" }` and/or `{ name: string }`
- Validate that the target user exists — 404 if not
- **Self-protection:** Admin cannot change their own role (prevents accidental lockout)
- Update the user record
- Return the updated user record

`DELETE /api/users/{user_id}` — Remove user access:
- Removes the user from the `users` table
- **Self-protection:** Admin cannot delete themselves
- The user's Supabase Auth account is NOT deleted — only their access to the SOP platform is revoked. Next time they try to sign in, they'll get "Access denied."
- Return 204 No Content

**Pydantic Schemas (api/app/schemas.py):**
Add to existing schemas:
- `UserCreate` — `{ email: str, name: str, role: UserRole }`
- `UserUpdate` — `{ name: Optional[str], role: Optional[UserRole] }`
- `UserResponse` — `{ id: UUID, email: str, name: str, role: str, created_at: datetime, updated_at: datetime }`
- `UserListResponse` — list of `UserResponse`

**Frontend — User Management Page (frontend/src/routes/settings.tsx):**

Replace the existing placeholder. This page is only accessible to admins (wrapped in `ProtectedRoute` with `requiredRole="admin"` from Phase 1.5b).

Layout:
```

│  User Management                    [+ Add]  │
├──────────────────────────────────────────────┤
│  Email               Name        Role    ... │

│  lasya@keells.com    Lasya B.    Admin   ✏️  │
│  ...                                         │
└──────────────────────────────────────────────┘
```

Features:
- Table showing all users with email, name, role (as badge), and action buttons
- Role badges colour-coded: Viewer (grey), Editor (blue), Admin (purple)
- "Add User" button opens an inline form or modal:
  - Email input (required)
  - Name input (required)
  - Role dropdown: Viewer / Editor / Admin
  - Save button → calls `POST /api/users`
  - Show success/error feedback
- Edit button (✏️) per row → inline editing of name and role dropdown
  - Save → calls `PATCH /api/users/{id}`
  - Cancel to revert
- Delete button (🗑) per row → confirmation dialog "Remove {name}'s access?"
  - Confirm → calls `DELETE /api/users/{id}`
  - Cannot delete yourself (button disabled or hidden for current user)
- Error handling:
  - Duplicate email → show "A user with this email already exists"
  - Network error → show generic error message
- TanStack Query for data fetching + mutations with cache invalidation

**Frontend Types (frontend/src/api/types.ts):**
The `AppUser` type from Phase 1.5a covers the response. Add:
- `UserCreateInput` — `{ email: string, name: string, role: 'viewer' | 'editor' | 'admin' }`
- `UserUpdateInput` — `{ name?: string, role?: 'viewer' | 'editor' | 'admin' }`

**Frontend API Functions (frontend/src/api/client.ts):**
Add:
- `fetchUsers()` — `GET /api/users`
- `createUser(data: UserCreateInput)` — `POST /api/users`
- `updateUser(id: string, data: UserUpdateInput)` — `PATCH /api/users/{id}`
- `deleteUser(id: string)` — `DELETE /api/users/{id}`
- `userKeys` query key factory: `userKeys.all`, `userKeys.detail(id)`
