# Phase 1.5b: Sign-in Page + Protected Routes

### Objective
Build the sign-in page with "Sign in with Microsoft" button, handle the OAuth callback, create a protected route wrapper that checks authentication and role, and update all existing routes to require authentication.

### Prerequisites
- Phase 1.5a complete ÔÇö Supabase client configured, useAuth hook working, GET /api/auth/me endpoint returning user data

### What to Build

**Sign-in Page (frontend/src/routes/login.tsx):**

A clean, centered sign-in page. This is the first thing users see when not authenticated.

Layout:
- Centered card on a subtle background
- App logo/name: "SOP Automation Platform" at the top
- Brief description: "Sign in to access Standard Operating Procedures"
- "Sign in with Microsoft" button with Microsoft icon
  - On click: calls `signInWithMicrosoft()` from `useAuthContext()`
  - Shows loading state while redirect is happening
- Footer text: "Only authorized users can access this platform"

Design:
- Full viewport height, centered vertically and horizontally
- Card: white background, subtle shadow, rounded corners, max-width ~400px
- Microsoft button: dark background (#2f2f2f), white text, Microsoft logo icon
- Clean, professional ÔÇö matches the rest of the app's aesthetic

This route must be accessible WITHOUT authentication (it's the public entry point).

**Auth Callback Handler (frontend/src/routes/auth.callback.tsx):**

Supabase redirects here after Azure AD authentication completes.

Behaviour:
- On mount, Supabase client automatically picks up the OAuth code from the URL hash/params
- Show a loading spinner: "Completing sign in..."
- The `useAuth` hook's `onAuthStateChange` listener detects the new session
- The hook fetches `GET /api/auth/me` to check if the user is in the `users` table
- If authorized Ôćĺ redirect to `/dashboard`
- If NOT authorized (403 from API) Ôćĺ redirect to `/login` with error state, show "Access denied ÔÇö your account is not registered. Contact your administrator."

This route must be accessible WITHOUT authentication (it's part of the auth flow).

**Access Denied Component (frontend/src/components/AccessDenied.tsx):**

Shown when a user authenticates via Microsoft but their email is not in the `users` table.

Content:
- Warning icon
- "Access Denied"
- "Your account ({email}) is not registered for this platform."
- "Contact your administrator to request access."
- "Sign out" button that calls `signOut()` and returns to /login

**Protected Route Wrapper (frontend/src/components/ProtectedRoute.tsx):**

A component that wraps route content and enforces authentication + optional role checks.

Props:
- `children: React.ReactNode` ÔÇö the route content to render
- `requiredRole?: 'viewer' | 'editor' | 'admin'` ÔÇö minimum role required (optional)

Behaviour:
- Reads auth state from `useAuthContext()`
- If `loading` Ôćĺ show a loading spinner (full page)
- If not authenticated Ôćĺ redirect to `/login`
- If authenticated but role check fails Ôćĺ show "You don't have permission to access this page" with option to go back to dashboard
- If all checks pass Ôćĺ render `children`

Role hierarchy for checks:
- `admin` > `editor` > `viewer`
- If `requiredRole` is `editor`, both `editor` and `admin` pass
- If `requiredRole` is `admin`, only `admin` passes
- If `requiredRole` is not specified, any authenticated user passes

**Route Updates:**

Update all existing routes to use ProtectedRoute:

| Route | Required Role | Notes |
|-------|--------------|-------|
| `/login` | None (public) | Accessible without auth |
| `/auth/callback` | None (public) | Accessible without auth |
| `/dashboard` | `viewer` | Any authenticated user |
| `/sop/:id` | `viewer` | Any authenticated user |
| `/sop/:id/procedure` | `viewer` | Any authenticated user |
| `/sop/:id/overview` | `viewer` | Any authenticated user |
| `/sop/:id/matrices` | `viewer` | Any authenticated user |
| `/sop/:id/history` | `viewer` | Any authenticated user |
| `/sop/new` | `admin` | Only admins can upload |
| `/settings` | `admin` | Only admins manage users |

Implementation approach ÔÇö wrap in the root or parent routes:
- `__root.tsx` ÔÇö wrap the `AuthProvider` around the entire app
- `sop.$id.tsx` (parent layout) ÔÇö wrap with `<ProtectedRoute requiredRole="viewer">`
- `sop.new.tsx` ÔÇö wrap with `<ProtectedRoute requiredRole="admin">`
- `settings.tsx` ÔÇö wrap with `<ProtectedRoute requiredRole="admin">`
- `dashboard.tsx` ÔÇö wrap with `<ProtectedRoute>`

**Navigation Update (frontend/src/components/Layout.tsx):**

Update the header/nav to show:
- User's name and role badge (from `useAuthContext()`)
- Sign out button/link
- Conditionally show nav items based on role:
  - "Upload" link Ôćĺ only for admin
  - "Settings" link Ôćĺ only for admin
- Dashboard link always visible for authenticated users

**Session Persistence:**

Supabase Auth handles session persistence automatically via localStorage. On page refresh:
1. `useAuth` hook checks `supabase.auth.getSession()` on mount
2. If session exists and is valid Ôćĺ fetches `/api/auth/me` Ôćĺ user is signed in
3. If session is expired Ôćĺ Supabase auto-refreshes the token (built-in)
4. If no session Ôćĺ user sees login page
