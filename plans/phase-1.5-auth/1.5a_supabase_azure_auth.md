# Phase 1.5a: Supabase Auth + Azure AD Configuration

### Objective
Configure Azure AD as an OAuth provider in Supabase Auth, install the Supabase client library in the frontend, and create auth utility hooks for managing authentication state.

### Prerequisites
- Phase 1 complete (Docker containers, API, React scaffold all working)
- Azure AD tenant access with permission to create app registrations
- Supabase project with Auth enabled

### Manual Setup (do these before running Claude Code)

These steps must be done manually — Claude Code cannot do them:

**Step 1: Azure AD App Registration (Azure Portal):**
- Go to Azure Portal → Azure Active Directory → App registrations → New registration
- Name: "SOP Automation Platform" (or similar)
- Supported account types: "Accounts in this organizational directory only" (single tenant)
- Redirect URI: leave blank for now (we'll add it in Step 2)
- After creation, note down:
  - **Application (client) ID** — this is the Client ID
  - **Directory (tenant) ID** — this is the Tenant ID
- Go to Certificates & secrets → New client secret → note down the **Value** (this is the Client Secret)
- Go to API permissions → Ensure `openid`, `email`, `profile` are granted (usually default)

**Step 2: Add Redirect URI in Azure (Azure Portal):**
- Go to your app registration → Authentication → Add a platform → Web
- Add this redirect URI (copy from Supabase Auth dashboard):
  `https://<your-supabase-project-ref>.supabase.co/auth/v1/callback`
- Save

**Step 3: Configure Supabase Auth (Supabase Dashboard):**
- Go to Supabase Dashboard → Authentication → Providers → Azure
- Enable the Azure provider
- Enter:
  - Azure Client ID (from Step 1)
  - Azure Secret Value (from Step 1 — use the Value, not the Secret ID)
  - Azure Tenant URL: `https://login.microsoftonline.com/<tenant-id>`
- Save

**Step 4: Get Supabase Keys (Supabase Dashboard):**
- Go to Supabase Dashboard → Settings → API
- Copy: **Project URL** → this is `VITE_SUPABASE_URL`
- Copy: **anon public key** → this is `VITE_SUPABASE_ANON_KEY`
- Copy: **JWT Secret** → this is `SUPABASE_JWT_SECRET`

**Step 5: Add yourself as admin (Supabase SQL Editor):**
```sql
INSERT INTO users (email, name, role)
VALUES ('your-email@cloudnavision.com', 'Your Name', 'admin');
```
Replace with your actual Microsoft/Azure AD email. This is needed to test sign-in from Phase 1.5b onwards.

**Step 6: Update .env file (in sop-platform/):**
Add these to your `.env` file:
```
VITE_SUPABASE_URL=https://<your-project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<your-anon-key>
SUPABASE_JWT_SECRET=<your-jwt-secret>
```
Also update `.env.example` with the variable names (no actual values).

### What to Build

**Install Dependencies (in frontend/):**
- `@supabase/supabase-js` — Supabase client library for auth

**Supabase Client Singleton (frontend/src/lib/supabase.ts):**
- Import `createClient` from `@supabase/supabase-js`
- Read `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` from `import.meta.env`
- Export a single `supabase` client instance
- This client is used for auth operations (signInWithOAuth, signOut, getSession, onAuthStateChange)
- NOT used for data queries — data still goes through FastAPI

**Auth Hook (frontend/src/hooks/useAuth.ts):**

A React hook that provides auth state to the entire app:

State:
- `user: SupabaseUser | null` — the authenticated Supabase user (has `email`, `id`)
- `appUser: AppUser | null` — the user record from the `users` table (has `role`, `name`)
- `loading: boolean` — true while checking initial session
- `isAuthenticated: boolean` — derived: `user !== null && appUser !== null`

Behaviour:
- On mount, call `supabase.auth.getSession()` to check for existing session
- Subscribe to `supabase.auth.onAuthStateChange` for login/logout events
- When a Supabase session exists (user authenticated via Azure), fetch the user's record from the `users` table via the API: `GET /api/auth/me`
  - If the API returns a user record → set `appUser`, user is authorized
  - If the API returns 403 (email not in users table) → user is authenticated but NOT authorized → sign out and show access denied
- Expose functions:
  - `signInWithMicrosoft()` — calls `supabase.auth.signInWithOAuth({ provider: 'azure' })` with redirect URL to `/auth/callback`
  - `signOut()` — calls `supabase.auth.signOut()`, clears `appUser`, redirects to `/login`

**AppUser Type (frontend/src/api/types.ts):**
Add to existing types:
```typescript
export interface AppUser {
  id: string;
  email: string;
  name: string;
  role: 'viewer' | 'editor' | 'admin';
  created_at: string;
}
```

**Auth Context (frontend/src/contexts/AuthContext.tsx):**
- Create a React Context that provides `useAuth` values to the component tree
- `AuthProvider` component wraps the app in `__root.tsx`
- All components can access auth state via `useAuthContext()` hook
- This is separate from the Zustand store — auth state belongs in React Context because it drives routing logic and needs to be available before any Zustand-dependent components render

**API Client Update (frontend/src/api/client.ts):**
- Update the fetch wrapper to include the Supabase JWT in the `Authorization` header
- On each API call: get the current session from `supabase.auth.getSession()`, extract the access token, add `Authorization: Bearer <token>` header
- If no session exists, redirect to `/login`

**Backend Auth Endpoint (api/app/routes/auth.py):**
- `GET /api/auth/me` — validate the JWT from the Authorization header, look up the user's email in the `users` table
  - If found: return the user record (id, email, name, role)
  - If NOT found: return `403 Forbidden` with message "Access denied — contact your administrator"
- This is the endpoint that bridges Supabase Auth (knows who you are) with the app's users table (knows if you're allowed in and what role you have)
